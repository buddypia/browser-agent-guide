## What this is

Browser Agent Guide is a **working Chrome Manifest V3 side-panel extension** (not a guide doc), plus a separate local Node daemon. The extension lets an LLM operate *any* web page through a **closed, deterministic verb registry** instead of free-form DOM access: the user chats in the side panel, the service worker collects page context and calls an LLM with Structured Outputs to get a fixed list of verbs + args, and the content script executes them deterministically.

The repo is **two independent npm projects** that are installed and tested separately:
- **Repo root** — the extension. Vanilla JS, ES modules, **no build step, no bundler, no linter, no CI**.
- **`daemon/`** — a separate ESM Node service (`bag-visual-feedback-daemon`) that bridges drawn-annotation screenshots from the browser to AI coding CLIs over MCP. Has its own `package.json` / `node_modules`.

UI strings are bilingual JP/EN by design (e.g. `お描き/Draw`); `README.md` is the authoritative English source and `README.ja/ko/zh.md` are translations.

## Extension architecture

Five MV3 contexts, each with a distinct job (they all share `chrome.runtime.onMessage`):

- **Service worker** (`background/service-worker.js`, `type:module`) — the orchestrator, holds no DOM. The **only** context that calls the LLM (`lib/ai-client.js`) and the only one allowed privileged APIs: `chrome.userScripts.execute`, `chrome.tabs.captureVisibleTab`, `chrome.offscreen`, `chrome.downloads`. Routes messages, auto-applies saved recipes on navigation, relays every page action to the content script.
- **Content script** (`content/content-script.js`) — a plain (non-module) IIFE injected into `<all_urls>`, guarded by `window.__AI_ADVISOR_INSTALLED__`. It **is the verb registry**: the `AI_VERBS` object (~40 verbs), the element resolver, deterministic affordance annotation (assigns stable `data-bag-id` like `button#3`), and the persistent user-annotation/drawing system. No LLM/network access of its own.
- **Side panel** (`sidepanel/sidepanel.js`) — the chat UI. Stateless about pages; talks only to the service worker via `chrome.runtime.sendMessage`. Per-page chat/prompt history in `chrome.storage.local`.
- **Options page** (`options/options.js`) — settings editor (AI provider/key, site rules, recipes JSON, daemon). Reads/writes the single settings blob via `lib/storage.js`.
- **Offscreen document** (`offscreen/offscreen.js`) — exists **solely** because the MV3 service worker has no DOM/Canvas; it composites annotations onto screenshots with `OffscreenCanvas` + Canvas 2D. Only one offscreen doc may exist (`ensureOffscreen()` guards creation); it ignores any message where `msg.target !== 'offscreen'`.

The `lib/` modules are the "brains" — pure-ish ES modules with no global state, consumed by the service worker and offscreen doc, hence unit-testable in Node:
- `lib/ai-client.js` — provider-agnostic LLM call, **always** Structured Outputs. `'openai'` and `'custom'` share `callOpenAICompatible` (json_schema strict); `'anthropic'` forces structure via a single tool + `tool_choice`; `'gemini'` uses `responseJsonSchema` + injected `propertyOrdering`. Each provider reads a *different* model field (`ai.model` / `ai.anthropicModel` / `ai.geminiModel`).
- `lib/prompt.js` — `buildSystemPrompt` assembles the (Japanese) system prompt from the live verb catalog + page affordances/targets + human annotations + a drawing workflow, and encodes the safety/determinism policy.
- `lib/storage.js` — single settings blob under `chrome.storage.local` key `aiAdvisorSettings`. `DEFAULT_SETTINGS` is the schema source of truth; `getSettings` deep-merges defaults per sub-object.
- `lib/site-matcher.js` — pure URL matching (`all/domain/page/prefix/regex`) deciding extension + recipe activation.
- `lib/recipe-merge.js` — pure learned-recipe merge: folds newly-learned actions into an existing recipe, preserves existing hand-edited actions, carries optional `when`/`waitFor`, and dedupes on `verb+args+when+waitFor`. Consumed by `rememberPageRule` in the service worker; unit-tested by `test/recipe-merge.test.mjs`.
- `lib/visual-feedback/compositor.js` — Canvas-2D-only annotation compositor.

### Main chat flow (page action), end to end
1. Side panel → `CHAT{tabId,text,history,rememberScope}` to SW.
2. SW `runChat`: `getSettings` → `COLLECT_CONTEXT` to content (returns verb catalog + affordances/targets + annotations) → `buildSystemPrompt` → `callAI`.
3. SW → `RUN_ACTIONS` to content; content executes each verb deterministically in document order (with a source-based blocklist — destructive verbs are rejected on the `'chat'` path).
4. SW `rememberSuccessfulChanges` persists successful recipe verbs as a learned site rule + recipe; on later navigation `syncTab`/`ACTIVATE` re-applies them once. **SPA internal navigation** (no full reload) is covered too: the content script detects the `location.href` change (`MutationObserver` + `popstate`/`hashchange` via `handleUrlChange`), resets `appliedRecipeSig`, and sends `SPA_NAVIGATED{url}` → SW re-runs `syncTab`. Recipe actions accept optional `waitFor:{selector,timeoutMs}` (defer until the element appears, 5s default; failure if it never does) and `when:{urlContains,selectorExists,selectorAbsent}` (skip unless matched). `runActions` evaluates them whenever present, but **only recipe (ACTIVATE) actions can actually carry them** — the chat-path Structured Outputs schema only emits `{verb,argsJson,reason}`, so chat-driven actions never have `when`/`waitFor`. Learned-recipe persistence (`mergeRecipeActions`, now in `lib/recipe-merge.js`) **carries `waitFor`/`when`** when an action has them and dedupes on `verb+args+when+waitFor` (so per-screen variants survive), but since the chat schema still doesn't emit them you **add** them by hand-editing the recipe JSON (Options → Recipes); once present they survive subsequent learning merges.

Visual-feedback (capture): the capture collects BOTH お描き (`kind:'drawing'`) and メモを残す (`kind:'note'`, non-empty text only) annotations (`collectVisualFeedbackData`; markers/buttons excluded; a note carries no shapes, so its bbox is the anchored element rect and only a callout+badge are burned in). SW captures the active target tab → offscreen composites the annotations onto the PNG → if the daemon is enabled, WebSocket push to `ws://127.0.0.1:8765/ws?token=…`; else `chrome.downloads` to `<browser download dir>/ai-inbox/<slug>/{shot.png,raw.png,annotation.json,memo.md}`. Manual capture refuses to run if the stored target tab is no longer active, because `chrome.tabs.captureVisibleTab(windowId)` captures the active tab in that window. The side panel displays the live target `tabId`/`windowId`/tab position; `annotation.json`, `memo.md`, and the daemon payload persist `tab:{tabId,windowId,index,active}`. The `<slug>` is `{localYYYYMMDD-HHMMSS}__{host}__{title}__{id}` (see `lib/slug.js`). On the download path the SW then resolves the **absolute** on-disk path via `chrome.downloads.search` (`DownloadItem.filename`), surfaces it in the side panel, and caches the discovered downloads dir; on the daemon path the SW sends that dir as `payload.downloadsDir` so the daemon can adopt it.

## Daemon architecture

One `http.Server` on one loopback port (default 8765) serves four surfaces: `/healthz` (JSON probe), `/mcp` (MCP Streamable HTTP), token-protected image GET routes `/shot/<id>.png` + `/raw/<id>.png`, and a WebSocket `upgrade` on `/ws` (extension push). The flow: extension pushes annotated PNG + metadata over WS → `store.js` `entryStore` takes it (**default `memory`: held in RAM only, NO inbox dir created**; `disk`/`hybrid` persist `<inbox>/<slug>/` via `writer.js`) → an AI CLI POSTs to `/mcp` and first calls `get_latest_feedback_context`, getting lightweight text/structured metadata (`tab`, `@agent:`, selector, testid, anchorLabel) without image tokens. It calls `get_latest_feedback_image` only when vision is needed, passing the context `id` as `contextId` plus a concrete `imageReason`, then gets the image as MCP vision content **plus** an absolute `file_path` (the image fallback invariant). Image tools refuse to return image content when the supplied `contextId` does not match the current entry.

- `src/index.js` wires the http server (`http.js`) + WS server (`ws.js`) + token; prints MCP/WS urls + token to **stderr** (stdout stays clean). Default `storage` is **`memory`** (no inbox; `--storage`/`BAG_VF_STORAGE` → `hybrid`/`disk` to use `<inbox>/<slug>/`); on `SIGINT`/`SIGTERM` it calls `entryStore.cleanup()` to drop any `memory`-mode OS-tmp materialization.
- `src/store.js` — the `entryStore` abstraction with three modes (`normalizeStorageMode`): **`memory` (default)** keeps captures in RAM (FIFO cap 50) and, when an image/`file_path` is actually requested, materializes **to a per-process OS-tmp dir** (`mkdtemp`, dropped on `cleanup()`), so the user's `<Downloads>/ai-inbox` is **never created**; `hybrid` is the same but lazily materializes into the real `<inbox>/<slug>/`; `disk` writes eagerly on every WS push (`writeEntry`). `list_feedback`/retention/cross-restart history exist only for `disk`/`hybrid`. The two image tools' Codex⇔Claude-Code parity (no `structuredContent`) is unchanged across all modes.
- `src/http.js` — Node built-in `http` (not express/hono). MCP is **stateless**: a fresh `McpServer` + `StreamableHTTPServerTransport` per POST, so all CLIs can hit one endpoint concurrently. 8MB body cap. Also serves token-protected `GET /shot/<id>.png` + `/raw/<id>.png` (constant-time `tokenEquals`; 401 on mismatch/missing; id restricted to slug charset so no traversal; served from `entryStore` so `memory`/`hybrid` `shot` streams from the in-memory buffer **without** materializing to disk). Per-POST it builds a `shotUrlFor(id,kind)` from `req.headers.host` and passes it to `createMcpServer`, so `*_context`/image tool text carry `shot_url`/`raw_url` (a dir-independent fetch target) alongside `file_path`. **The emitted URL is token-LESS on purpose** — `/mcp` is unauthenticated, so embedding the write-capable token into a read-only MCP response would be a privilege-escalation path; the consumer appends `?token=`. URL emission is gated on a token being configured (so `createHttpServer({inboxDir})` in tests stays URL-free, and the URL only appears when the route is actually usable). The URL string is built by `src/image-url.js` (`imageUrlFor(host,id,kind)`) — a single pure helper shared by this `shotUrlFor` and the `ws.js` ack so the format can't drift from the route regex. The image tools materialize via a **best-effort** `materializeForImage` (server.js): image bytes always come from memory (`readEntryImageBuffer`), the disk `file_path` materialize is wrapped in try/catch, and if it fails (read-only inbox / disk full) the tool still returns the inline image + `shot_url` and `buildEntryText` omits the now-bogus `file_path` line — so an image request is never lost to a disk error (mirrors `/shot`, which never needs disk to hand back bytes).
- `src/ws.js` — token-checked WS receive (constant-time compare; 401 + destroy on mismatch). 64MB payload cap (composite PNG base64 is large). The `ack` carries a token-less `shotUrl` (and `rawUrl` when the push included raw) built via `imageUrlFor(req.headers.host, …)`, so the extension side panel can surface a dir-independent fetch URL for the just-saved capture (consumer appends `?token=`; in `index.js` the WS and the image route share one token, so the advertised URL is always reachable).
- `src/server.js` — MCP server `bag-page-feedback` (renamed from `bag-visual-feedback`) with 5 tools, now **modality-neutral named**: `list_feedback`, `get_latest_feedback_context`, `get_feedback_context` (text/HTML/a11y, **no image**), `get_latest_feedback_image`, `get_feedback_image` (vision). The deprecated `*_visual_feedback*` aliases were **removed** — only the 5 new names are registered (`tools/list` returns exactly 5; `registerWithAlias` is gone). `urlContains` / `titleContains` filters are optional case-insensitive substring matches; `tabId` / `windowId` filters disambiguate same-URL tabs using the Chrome-session-scoped tab metadata. The two image tools require `contextId` + `imageReason` so callers cannot skip the lightweight `@agent:`/selector/**html** pass by accident.
- `src/inbox.js` / `src/writer.js` — read (newest-first scan, skips `done/`) and write (server-side slug via `src/slug.js` from `capturedAt`+`url`+`title`, tmp→rename atomic, 0600/0700, rollback on failure) sides. `inbox.js` also owns `resolveDownloadsDir()` (Win32 registry / Linux XDG / macOS `~/Downloads`) used by `defaultInboxDir()`.
- **`inboxDir` is runtime-mutable**: `index.js` holds `inboxState.dir` and passes a getter (not a string) to `http.js`/`ws.js`. When the extension reports `downloadsDir` over WS, `index.js` `adoptDownloadsDir` switches the inbox to `<downloadsDir>/ai-inbox` — UNLESS `--inbox`/`BAG_VF_INBOX` pinned it. So tests/callers may pass `inboxDir` as a string **or** a `() => dir` function.
- `src/token.js` — shared secret at `~/.bag-vf/token`; precedence flag > env > persisted file > newly generated.

## Conventions & gotchas

- **The verb registry is a CLOSED set.** The LLM can only emit verbs that exist in `AI_VERBS` (`content/content-script.js`), enforced by an enum in `buildResponseSchema` (`lib/ai-client.js`), generated from the live catalog returned by `COLLECT_CONTEXT`. Add a capability = add one verb entry; it auto-flows into both the prompt and the schema.
- **Action args travel as `argsJson` (a JSON *string*), not a nested object** — deliberately, because strict Structured Outputs requires every property required + `additionalProperties:false`, impossible for verbs with heterogeneous arg shapes. `callAI` re-parses `argsJson`. Don't try to put a typed args object in the schema.
- **`compositor.js` must stay Canvas-2D-only** — no SVG/`foreignObject`/`createElementNS`/`new Image`/`drawImage` in the module body (SVG foreignObject taints the canvas → `convertToBlob()` throws SecurityError). A banned-token scan in `compositor.test.mjs` **fails the build** if you add them. `drawImage` is allowed only in `offscreen.js` (the screenshot background).
- **Coordinates are layered**: drawings stored as 0..1 fractions of the anchored element's rect (survive reflow/scroll); at composite time `factor = dpr × outputScale` converts CSS px → output px; `computeOutputSize` caps the long edge at 2000px (vision APIs reject larger). Apply `factor` uniformly to every coordinate and line width.
- **Determinism is load-bearing**: temperature defaults to 0, affordance `aiId`s assigned in document order, recipes re-applied on revisit (and on SPA internal navigation via `SPA_NAVIGATED`→`syncTab`). Annotations persist per `origin+pathname` under `chrome.storage.local` key `aiAdvisorAnnotations`.
- **API key lives in `chrome.storage.local`, NOT `sync`** (it's a secret) — keep it that way. Always read/write settings via `lib/storage.js` so defaults merge.
- **Two recipe allow-lists must stay in sync**: `SAFE_RECIPE_VERBS` (`options/options.js`) and `RECIPE_VERBS` (`background/service-worker.js`) — both = `injectHtml/injectCss/injectScript/outlineElement/injectButton/injectPanel`. Only these get persisted and auto-replayed.
- **Security boundary**: high-risk verbs (`setStyle`, `removeElement`, `defineMarker`) are hidden from chat and rejected on chat/auto-recipe paths; page text/attributes are treated as **untrusted** (prompt-injection defense in `prompt.js`). Don't widen verb exposure casually.
- **`injectScript` runs via the User Scripts API in the service worker**, not the content script (content sends `EXECUTE_USER_SCRIPT` to SW → `chrome.userScripts.execute`). Requires Chrome 135+ and the user manually enabling "Allow User Scripts" in `chrome://extensions`.
- **If you add a source JS file**, add it to the `check:js` file list in `package.json` (an explicit `node --check` loop — no glob). Packaging (`npm run zip` → `package-extension.sh`) uses an **exclude-list** (`zip -r . -x …`), so new source files are bundled automatically; only new dev/test/doc directories need a new `-x` exclude there.
- **Glossary = domain SSOT (`glossary/<ctx>/<id>.md`), enforced like the verb registry**: natural-language tasks anchor to a term `id`. Each entry's frontmatter is the machine-readable truth (definition, `owner`, `status`, `source_refs`/`code_refs`/`api_refs`/`db_refs`, `last_verified`, `confidence`); the schema lives in `glossary/_schema.md` and the machine SoT is `scripts/glossary/validate.mjs`. **The "edit code → update the doc" guarantee is the staleness gate**: load-bearing code is wrapped in `// @term: <id>` … `// @endterm` guard regions, and `scripts/glossary/check-staleness.mjs` fails if a changed guard region's bound term did not bump `last_verified` in the same diff. Day-to-day: `npm run glossary:staleness` → if it flags a term, reconcile the entry then `npm run glossary:touch <id>`. `npm run check:glossary` validates the whole set; `npm run glossary:trace` regenerates the (gitignored, generated) `glossary/TRACEABILITY.md` RTM. Enforcement is layered: optional pre-commit hook (`bash scripts/glossary/install-git-hook.sh`, bypass `git commit --no-verify`) → `npm run check` → CI (`.github/workflows/glossary.yml`, the backstop). **Before starting a task, read only the relevant term entries and restate `id` / definition / `code_refs` / `confidence`, normalizing any `deprecated_terms` to the canonical name** — this is the anti-misunderstanding protocol. Full runbook: `docs/glossary-mechanism.md`. Deliberate non-goals: CI never judges semantic correctness (human/owner review does), definitions are human-written (only traceability is generated), and `TRACEABILITY.md` is never committed.
- **The slug generator is DUPLICATED in two independent npm projects with no shared import**: `lib/slug.js` (extension) and `daemon/src/slug.js` (daemon) must stay **byte-identical** — `test/slug.test.mjs` imports both and fails if they drift. Folder name = `{localYYYYMMDD-HHMMSS}__{host}__{title}__{id}`; id is a **synchronous** FNV-1a hash (NOT `crypto.subtle`, which is async-only in the SW). The folder name **is** the daemon entry `id` (`inbox.js` `id: d.name`), so changing the scheme invalidates previously-emitted ids; old timestamp-named folders still list fine (scan reads arbitrary names). Embedding host/title in the name is a human/AI **skim aid only** — `annotation.json` stays the authority for `urlContains`/`titleContains` (the sanitized name is lossy, so name-matching would false-negative).
- **The `chrome.downloads` save path is the browser's configured download dir, NOT necessarily `~/Downloads`** (user-moved, Edge/Brave, or "ask where to save"). Never assume `~/Downloads/ai-inbox` for the FILE fallback: the SW resolves the real absolute path (`chrome.downloads.search` → `DownloadItem.filename`) and the daemon auto-detects + adopts the reported `downloadsDir`. With the daemon ON the path is owned end-to-end (no mismatch); the mismatch only exists on the daemon-OFF fallback. **The inbox path is NOT a required handoff channel when the daemon is ON**: the image rides base64 over WS into the daemon and back out as inline MCP image content; the CLI addresses captures by `id`, not by path. `adoptDownloadsDir`/`downloadsDir` is a consistency convenience (optional — the WS push succeeds without it), and the daemon-OFF `chrome.downloads` fallback is the only place path coordination is genuinely needed. The token-protected `GET /shot/<id>.png` route makes even that addressing dir-independent (fetch by `id`, no filesystem path); the daemon WS `ack` carries that token-less URL and the side panel renders it as a "画像URL/Image URL" line (`capture.imageUrl`, append `?token=`) so the user/AI can reach the just-saved capture without knowing the inbox path.
- **Three test runners, easy to confuse**: Playwright Test runs `*.spec.mjs` (`testMatch` excludes `.test.mjs`); `npm test` runs `test/anchor.playwright-cli.mjs`, which shells out to `playwright-cli run-code` rather than importing Playwright directly; compositor `*.test.mjs` (root) is a bare node script; daemon `*.test.mjs` run via `node --test`.
- **Daemon MCP registration uses a user-chosen alias (`bag_page_feedback`) and a different JSON key per CLI**: Claude Code / Codex use `url`, Antigravity uses `serverUrl` — all pointing at `http://127.0.0.1:8765/mcp` (see `daemon/README.md`). Auth is WS-only; `/mcp` relies on loopback binding. **The registration alias (the `bag_page_feedback:` prefix the model types) is USER-CHOSEN in their CLI config and is NOT the same as the server's internal `name`** — renaming the server `name` (`bag-visual-feedback`→`bag-page-feedback`) does not change a user's existing alias, only what `claude mcp add` would suggest. The docs/skill use `bag_page_feedback`; a legacy `bag_visual_feedback` registration must be re-pointed (`claude mcp remove bag_visual_feedback && claude mcp add --transport http bag_page_feedback …`), and the skill `preflight.sh` now detects only `bag_page_feedback`. Both the **tool names** (after the colon) were renamed AND the deprecated `*_visual_feedback*` tool aliases were removed (new names only).
- **MCP tools are modality-neutral named + carry the annotated element's HTML (schema v1)**: the capture now stores each annotated element's `outerHTML` (≤8KB, `truncated:true` past that) + compact a11y (role/name/level/state) in `annotation.json` items (`bag.visual-feedback/v1`; `collectVisualFeedbackData`→`buildAnnotationJson`). The **context tools** (`get_latest_feedback_context`/`get_feedback_context`, formerly `*_visual_feedback_context`) surface `html`/`a11y` in both text and `structuredContent` — so "fetch the HTML element I left a memo on, no image" is a first-class, image-token-free path. The **image tools** (`get_latest_feedback_image`/`get_feedback_image`) are unchanged and still return `{content}` only (Codex parity). The old `*_visual_feedback*` tool aliases were **removed** — only the 5 new names exist (`registerWithAlias` gone; `tools/list` returns 5). v0 entries (no html/a11y) normalize to `null` (`normalizeHtmlCapture`/`normalizeA11yCapture` in `inbox.js`). `.claude/skills/bag-workflow/**` and the generated `docs/mcp-*.html` diagrams are synced to the new tool names + the `bag_page_feedback` alias (its `preflight.sh` detects only `bag_page_feedback`).
- **Image MCP results must NEVER carry `structuredContent` (Codex⇔Claude Code parity invariant)**: OpenAI Codex CLI drops the *entire* `content[]` — image block included — whenever a tool result also has `structuredContent` (codex#10334, OPEN), surfacing only `structuredContent`; Claude Code does not. So the two image tools (`get_latest_feedback_image` / `get_feedback_image`, `server.js`) return `{content}` ONLY; `structuredContent` appears solely on the three context tools + the disambiguation branch (none of which carry an image). A regression guard locks this so a future edit can't silently blind Codex: `assertImageHasNoStructured` in `daemon/test/mcp.test.mjs` (single/by-id/filtered/mixed-banner branches) + the same assert on the materialize-fail path in `image-fallback.test.mjs`. **Do NOT "fix" Codex by duplicating image bytes into `structuredContent`** — that re-triggers the very `content[]` drop. Codex base64 vision additionally needs its **rmcp client** (Streamable HTTP routes through rmcp) and, on builds where inline base64 isn't shown (codex#4819), its built-in `view_image` against the always-present `file_path`/`shot_url` — so that text channel is **load-bearing for Codex, not a mere fallback** (see `daemon/README.md` Codex section). The disambiguation result also mirrors its warning prose into `structuredContent.disambiguation.message` so Codex (which surfaces only `structuredContent`) reads the same "別案件を掴ませない" guidance — **but still emits NO top-level `id`** (the laundering-prevention invariant stays intact; `mcp.test.mjs` re-asserts `structuredContent.id === undefined`).
- **The MCP `{type:'image'}` block carries a COMPACT inline variant, NOT the full-res PNG (Claude Code output-token-cap fix)**: Claude Code tokenizes an MCP image's base64 as **text** against `MAX_MCP_OUTPUT_TOKENS` (default **25,000**) and **hard-errors + drops the image** on exceed — unlike text, images get no disk-persist fallback; only raising the env var helps (verified: code.claude.com/docs/en/mcp, claude-code#9152/#27869/#31208). A full-res 2000px PNG (~1–4MB base64) always blows it. Fix = the extension's `offscreen/inline-encode.js` `encodeInline` produces a second **WebP (JPEG fallback)** variant iterated down to a **~12KB budget** (`INLINE_BYTE_BUDGET`), shipped over WS as `payload.image.inline`+`inlineMime`; the daemon persists it as `shot.inline.<ext>` next to `shot.png` (disk/hybrid, survives restart) and holds `inlineBuffer`+`inlineMime` in RAM (memory mode, the default). `buildEntryContent` (`inbox.js`, the ONLY `{type:'image'}` site) prefers the inline bytes; if absent it falls back to full-res `shot.png`; if the chosen bytes exceed **`INLINE_MAX_BYTES` (14KB)** it **OMITS the image** (text-only with `file_path`/`shot_url`) so the cap is structurally un-blowable even for legacy/oversized entries. **The budget is load-bearing — do NOT raise it toward 30KB** (the 0.5 tok/char ratio that justified 30KB was refuted; real high-entropy base64 is ~0.94–1.0 tok/char on Opus 4.8, so >~16.4KB risks the cap). `file_path`/`shot_url`/`/shot/<id>.png`/`chrome.downloads` keep serving **full-res PNG** (Codex `view_image` + human download depend on it; the 2000px vision cap is unchanged). `inlineMime` is read from the actual produced `blob.type` and carried end-to-end (never hardcoded) so Claude's `media_type` passthrough can't mis-sniff (codex#39146). The inline/omit branches still return `{content}` ONLY (no `structuredContent`) — `assertImageHasNoStructured` covers them. The lossy encode lives **only in `offscreen/inline-encode.js`** (a sanctioned `drawImage`/`convertToBlob` site); `compositor.js` stays untouched, so its Canvas-2D-only scan can't trip.
- **The shared `~/Downloads/ai-inbox`** can return another project's last capture, so it is now **disambiguated + gardened** daemon-side (still: pass `urlContains`/`titleContains` to scope, and pass `tabId`/`windowId` when same-URL tabs are possible; a non-matching filter returns text-only on purpose):
  - **disambiguate-latest** (DEFAULT ON; `server.js` bare-branch + `peekDistinctRecent` in `inbox.js`): a BARE `get_latest_feedback*` call (no `urlContains`/`titleContains`) that sees **≥2 distinct hosts** within `latestWindowMs` (default 90m, **`capturedAt`-anchored** — file mtime collapses under git/rsync/download-fallback) returns a candidate list instead of one capture: NO image, and **NO top-level `structuredContent.id`** (so a foreign id can't be echoed back to launder an image; the image tool only honors a `contextId` matching an in-window candidate). Single-project inboxes are byte-identical to before; `--latest-window-min`/`BAG_VF_LATEST_WINDOW_MIN` tunes the window.
  - **retention sweep** (DEFAULT OFF, opt-in `--retention on`/`BAG_VF_RETENTION`; `daemon/src/retention.js` wired in `index.js` startup+unref'd-interval+onSaved+on-adopt): archives stale (past `maxAge`, default 14d) and over-cap same-`{host}__{title}`-family (keep newest `maxPerFamily`, default 5) entries into `<inbox>/done/` via **atomic rename, never delete** (read path unchanged — `listEntries` skips `done/`), **never within the `grace` window** (default 30m; `maxAge`/`doneTtl` clamp ≥ grace), purging `done/` after `doneTtl` (default 7d). `findEntry` now resolves an archived id from `done/` for by-id/image fetches while latest/list still exclude `done/`. No slug-scheme change, no 6th MCP tool. **GOTCHA: `familyKey` must strip BOTH the leading `{stamp}` and trailing `{hash}` of `{stamp}__{host}__{title}__{hash}`** — stripping only the hash (as the first design draft did) leaves the per-generation stamp in the key and makes the family cap a silent no-op.
- **UI is bilingual JP/EN**; keep new user-facing strings bilingual. The anti-slop UI quality workflow (`docs/ui-quality-workflow.md`) requires generated UI changes to be backed by a repeatable Playwright/axe test or screenshot, verified at side-panel width.
- **The Stop-time worktree staleness guard measures a *never-refreshed* local `origin/main` ref** (`worktree-shipping-guard.mjs` `measureStaleness`, network-free by design — `worktree-new.mjs` fetches only at creation) and only flags `behind>20`/`age>7d`. So it is **blind to remote `origin/main` divergence that lands *after* worktree creation**, and the `behind` count never predicts an upstream file *deletion/rewrite* that collides with your change (a 2-commit drift can still be a guaranteed modify/delete conflict). Before shipping a long-lived worktree in this multi-session-concurrent repo, `git fetch origin main` then `git diff --name-status origin/main...HEAD`, and re-scope if upstream removed/changed files your work depends on. See `docs/retros/retro-2026-06-20-worktree-base-remote-divergence.md`.
- **Governance hooks silently fail OPEN on a removed dependency** — `safeHookMainWithProfile` catches a hook's load error and passes through, so a deleted/renamed import inertly disables a guard with no signal (the `orphaned-guard-trigger` class). `node .claude/scripts/check-hook-refs.mjs` lints every `.claude/hooks` + `.claude/scripts` file so a dangling relative import or concrete `.claude/…*.mjs` path reference fails loudly; its test (in the `.claude/scripts/lib/__tests__` suite) asserts the live tree has zero dangling refs. Run the governance suite (`node --test .claude/scripts/lib/__tests__/*.test.mjs`) before shipping `.claude` changes. The complementary regex-trigger-liveness vector (a pattern naming a removed command) is covered by per-guard contract tests (`pre-ship-review-guard.test.mjs`, `commit-guard.test.mjs`, `destructive-git-guard.test.mjs`) that assert canonical commands still deny/pass. See `docs/retros/retro-2026-06-20-orphaned-ship-gate.md`.

- **`bag-workflow`** (`.claude/skills/bag-workflow/`) — project-scoped, **user-invoked** skill (`/bag-workflow [urlContains]`, `disable-model-invocation: true` so this side-effecting flow never auto-fires). Its job is to **teach a browser-operation workflow to the AI and have it carried out from the cues the user leaves on the page** — お描き is just ONE cue, not the whole story (other cues: live DOM/a11y, extension `data-bag-id`, `@agent:` markers, recorded steps). ## Git workflow

- **GitHub Flow** — never commit straight to `main`. Branch from `main` (`feat/…`, `fix/…`), commit, push, open a PR, then merge.
- **Commit messages in English**, imperative mood (e.g. "Add @agent: marker lint"); subject ≤ ~72 chars, body explains the *why* when non-trivial.
- **Shipping goes through `gh` directly** (the `create-pr` skill + `ops.mjs` were removed in the brief2dev cleanup): commit in the worktree (`git -C <wt>`), let `agent-worktree-guard` brief the completed work and ask whether to create a PR, then after the human answers Yes/進行 run `agent-worktree-guard confirm-pr --confirmed` before `gh pr create`. Before `gh pr merge` (the irreversible main merge), present the 7-section Pre-Ship Human Review Panel, get explicit user confirmation, and run `node .claude/scripts/mark-pre-ship-confirmed.mjs <branch> --quality <label>` (labels: `agent_go` / `self_review_pass` / `trivial_skip`). `pre-ship-review-guard` (PreToolUse) **blocks `gh pr merge` until that marker is fresh** (10 min) — run `gh pr merge` from the worktree so it infers the branch from cwd (else pass `--staged`). `agent-worktree-guard` additionally blocks `gh pr create/merge` before PR confirmation and refuses `cleanup --confirmed` until PR merge is recorded (automatic after successful `gh pr merge`, or `agent-worktree-guard mark-merged <path>` after manual verification). The `agent-worktree-guard` CLI is the wrapper at `scripts/agent-worktree-guard/agent-worktree-guard` — the `.claude/hooks/agent-worktree-guard.mjs` entry only dispatches `hook` subcommands (`pre-tool`/`post-tool`/…), so `confirm-pr`/`mark-merged`/`cleanup`/`status` must go through the wrapper, not the `.mjs`. `cleanup` resolves each worktree's owning session from its `.tmp/.agent_worktree_owner.json` marker, so `cleanup --confirmed` run from the **main checkout** still removes merged worktrees even when the shell can't resolve the original session id (it cross-checks `git worktree list` against per-session ledgers; it stays a no-op **and warns** when a registered worktree is not yet merged — the silent `cleaned 0` that previously orphaned merged worktrees is gone). **`cleanup` also tidies what the worktree left behind**: after `git worktree remove` it force-deletes the now-merged local branch (`git branch -D`, never `main`), **deletes the merged remote branch** (`git push origin --delete`, best-effort: gated on `origin` being configured + `ls-remote` showing the ref still exists, so a branch already removed by `gh pr merge --delete-branch` is a no-op `already-absent`; offline/auth failure is a non-fatal `failed` that just warns + asks you to verify on GitHub, never aborting the local cleanup), and drops only the stashes based on that exact branch (`WIP on <branch>`/`On <branch>`), reporting each on stderr + the `cleaned N worktree(s)` summary (`branches deleted: N, remote branches deleted: N, stashes dropped: N`) — stashes on other branches are never touched. The `# PR 承認後の完了報告` template's `## 2. Cleanup 状況` surfaces all four results (Local worktree / Local branch / Remote branch / stash), and the REVIEW.md scaffold's `### 承認後の Cleanup 手順` carries the same post-approval checklist. The `agent-worktree-guard` Stop/PR briefing (shared by Claude + Codex + Antigravity) **surfaces each worktree's full 11-section `REVIEW.md` approval review** (承認依頼 / 状態サマリー / コミット情報 / PR Draft / 修正ファイル / 変更内容 / なぜ / トレードオフ / リスク・Rollback / 残タスク / セッション改善); if `REVIEW.md` is absent it points at the scaffold. Separately, `worktree-review-report-guard` (Stop) requires a HEAD-stamped 11-section `REVIEW.md`, and `agent-worktree-guard cleanup --confirmed` prints `# PR 承認後の完了報告` after merge cleanup.
